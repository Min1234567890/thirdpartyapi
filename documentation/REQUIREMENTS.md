**Version**: 3.0 (.NET Core)
**Date**: 18 June 2026
**Status**: Implemented & Tested (26/26)

---

## 1. Background

Jurong Port SmartPass uses Techsphere VP-II X hand vascular pattern scanners at
gate lanes. The VPX communicates through a database command queue (TTask table)
managed by XAgent — there is no direct API to the hardware.

The ThirdpartyAPI replaces physical MIFARE cards with QR codes. A QR code provides
an 8-hex CSN. The API derives all VPX identifiers (PIN, DevicePIN) from this CSN
using the same Wiegand pipeline the card system uses, maintaining full backward
compatibility with gate lane hardware.

## 4. Functional Requirements

### R1 — Enrolment (POST /api/enrol)

Input: idType (NRIC/FIN/PASSPORT) + idNumber only. No CSN required.

#### Assumption

**CSN is pre-assigned in JCMS before enrolment.** The API reads the CSN from
`JC_CARDDTL.CSN_No` — it does NOT generate, derive, or write the CSN.

#### Card Classification

The API looks up ALL cards for the person (any status) and categorizes:

| Category | Criteria |
|---|---|
| **Target card** | STATUS_CD IN (USE, SUS, SUSV) AND CSN_No IS NOT NULL |
| **Old enrolled** | CSN_No IS NOT NULL, different card (any status) |

#### Enrolment Paths

| Scenario | Action | TTask Type |
|---|---|---|
| No cards found | `USER_NOT_FOUND` | — |
| No active card with CSN assigned | `FAILURE` | — |
| DevicePIN already in TUser | `ALREADY_ENROLLED` | — |
| Target card + NO old enrolled card | **New enrolment** | 159 |
| Target card + old enrolled card exists | **CopyUser**: copy template old→new | **137** |

### R4 — CSN & DevicePIN

| # | Requirement | Detail |
|---|---|---|
| R4.1 | CSN generated from JCMS card serial (SHA256 → 32-bit hex) | Deterministic — same card always same CSN |
| R4.2 | CSN = 8 hex chars | Same format as MIFARE card UID |
| R4.3 | DevicePIN = 52 chars per SDK §1.4.2 | Derived by API: "03"+"22"+Wiegand48 |
| R4.4 | DevicePIN passed to VPX in TTask InputData | VPX uses this as the user key |
| R4.5 | PIN generated by VPX/XAgent | Stored in TUser.PIN; read back by API |
| R4.6 | Same card serial always same CSN/DevicePIN | Deterministic hash + Wiegand pipeline |

| DI | Built-in Microsoft.Extensions.DependencyInjection |

## 6. Constraints

- Must not change gate lane code
- Must not change VPX firmware
- Must use existing XAgent
- Must produce card-compatible DevicePINs
- Must use existing HVPRX/SmartPass databases

## 7. Out of Scope

- Authentication (sandbox)
- Rate limiting
- TLS (internal network)
- Card reading/writing (QR code replaces)
- VPX firmware modification
- New databases
